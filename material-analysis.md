# 和他一起看材料的流程

> YouTube / B 站 / Twitter / 微信 / 小红书 / PDF / 网页——这些来源都有专门的提取脚本，提取质量和完整度比通用 WebFetch 好得多。收到链接直接进入下面的流程。

你从 SKILL.md 的"上课开始前"被引到这里，因为他给了一个外部链接，或者提到了之前一起看过的材料。

这个流程的核心：**外部材料是学习的输入，不是学习本身。** 你的工作不是帮他"总结"一个视频或文章，而是用这份材料作为素材，陪他把里面的东西真的搞懂、变成他自己的理解。

---

## 材料缓存

所有提取过的材料都存在 `~/.claude/skills/feynman-tutor/materials/` 里，避免重复提取和重复分析。

### 目录结构

```
materials/
├── INDEX.md                    # 材料库索引
└── <material-id>/              # 每个材料一个目录
    ├── meta.md                 # 元数据（URL、标题、来源、时长、提取日期）
    ├── content.md              # 完整原文（字幕/文章全文）
    └── analysis.md             # 分析报告 + 讨论进度追踪
```

### material-id 命名规则

- YouTube：`yt-<video-id>`（如 `yt-kwSVtQ7dziU`）
- B 站：`bili-<BV号>`（如 `bili-BV1xx411x7xx`）
- X/Twitter：`tw-<tweet-id>`（如 `tw-2039826228224430323`）
- 微信公众号：`wechat-<URL的SHA256前8位>`（如 `wechat-5a3b9c1e`）
- 小红书：`xhs-<note_id前8位>`（如 `xhs-6742a1b3`）
- 网页：`web-<URL的SHA256前8位>`
- PDF：`pdf-<URL或文件路径的SHA256前8位>`（如 `pdf-a3f1c92b`）

## 阶段 A：把材料拿到手

### A1 检查缓存

1. 从 URL 推导 material-id（如果他没给 URL 而是提到"上次那个视频"，根据描述在 INDEX.md 里找匹配）
2. 读 `~/.claude/skills/feynman-tutor/materials/INDEX.md`（没有这个文件说明是第一次用材料系统，跳 A2）
3. 查这个 material-id 是不是已存在

**缓存命中** → 加载 `materials/<id>/meta.md` 和 `materials/<id>/analysis.md`，跳到 **C-续读**。content.md 先不加载——只在需要深入某个话题时按行范围按需读。

**缓存没命中** → 继续 A2。

### A2 提取内容

运行提取脚本（脚本自带隔离 Python 环境，首次运行会自动装依赖）：

```bash
py -3 ~/.claude/skills/feynman-tutor/scripts/run.py "<URL>"
```

**提取失败时**（脚本返回的 JSON 里 `error` 不为 null）：
- 把脚本返回的错误信息**原样给他看**
- 如果错误信息里有 cookie 设置引导，帮他完成
- 给替代方案：他可以手动粘贴内容，或者提供其他来源
- **绝对不要在提取失败时凭训练数据的记忆声称自己了解内容**

**提取成功** → 继续 A3。

### A3 缓存提取结果

1. **建目录**：`materials/<material-id>/`

2. **保存 meta.md**：
```markdown
---
material_id: <id>
title: "<标题>"
url: <原 URL>
source: youtube | bilibili | twitter | wechat | xhs | web | pdf
extracted_date: <今天日期>
duration_seconds: <时长，如果有>
word_count: <字数估算>
---
```

3. **保存 content.md**：提取到的纯文本写进来，每行约 200-500 字符（按句子边界断行），方便后续按行范围读。保留原始段落结构。有时间戳的话在段落开头标注。

4. **更新 INDEX.md**：在索引表追加一行。讨论进度初始为 `0/<总话题数>`。

继续 B。

---

## 阶段 B：检查学习笔记

读 `~/.claude/skills/feynman-tutor/notes/INDEX.md`，关注点是：

1. 扫一下材料内容大致涉及哪些知识领域和主题
2. 在笔记索引里查相关主题
3. 找到了就读笔记，重点看：
   - **他已经通了的部分**：材料里涉及这些的地方可以快带过
   - **当前边界**：材料里触及边界附近的内容是教学重点
   - **还需要加强的**：材料正好覆盖了就优先深入
   - **对他有效的类比**：讨论时复用已经验证过有效的解释方式

---

## 阶段 C：材料分析报告（首次分析）

给他一份结构化的分析报告。目的是帮他在深入之前先建立全局认识。

对材料做**全量分析**——完整读 content.md，识别所有可独立讨论的话题，为每个话题标注在 content.md 中的行范围。

**超长材料（content.md > 500 行）的全量分析策略：**

1. 分段扫描：每次 Read 约 500 行，从头到尾逐段覆盖，不跳过
2. 每段记录发现的话题及行号范围
3. 全部扫完合并为完整话题地图
4. 覆盖验证：话题地图的行范围之并集应覆盖 content.md 至少 85% 的内容行——尾部有大段未覆盖说明扫描不完整，需要补读

**报告详细度自适应**：报告结构应和材料复杂度匹配。话题数 ≤3 的短材料，知识地图和推荐学习路径可以合并或省略——直接展示话题列表问他想深入哪个。

### 报告结构

```
📖 材料分析报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

来源：[YouTube / B 站 / 网页 ...]
标题：[材料标题]
时长 / 篇幅：[X 分钟 / 约 X 字]

---

## 全文主旨
[2-3 句话概括]

| # | 话题 | content.md 行范围 | 状态 |

| 1 | [话题名] | [行范围] | ⬜ 未讨论 |
| ... | | | |

## 知识地图

- [材料里涉及的、他已经懂的内容]

🔶 当前边界附近（推荐优先深入）：
- [材料里触及他知识边界的内容]

- [材料里他完全没碰过的内容]

## 各话题详细分析

**核心论点**：[简要概括]

```

### 报告生成后立即保存

完整报告写入 `materials/<id>/analysis.md`。顶部加 YAML frontmatter：

```yaml
---
material_id: <id>
title: "<标题>"
url: <URL>
source: <来源>
extracted_date: <日期>
duration_seconds: <时长>
total_topics: <话题总数>
discussed_topics: 0
last_discussed: <日期>
---
```

analysis.md 同时承担分析报告和进度追踪——话题地图里的"状态"列就是进度追踪。

**没有学习笔记时**：知识地图部分改成简要的内容分层（基础概念 / 进阶 / 前沿或有争议的）。

**报告要精炼**：这是导航，不是论文。每个条目一两句话。

## 阶段 C-续读：加载已有分析

缓存命中（他在新一次对话中回到同一份材料）时走这个流程，代替 B 和 C：

1. **加载 analysis.md**：拿到完整的话题地图和讨论进度。这个文件里有首次分析时的全部信息——全文主旨、各话题分析、知识地图、推荐路径。

2. **对比当前笔记**：读 notes/INDEX.md，看他在相关主题上有没有新进展（通过笔记的 last_updated 判断）。有的话可能会改变某些话题的优先级或连接点。

3. **给他看续读版报告**：

```
📖 材料续读报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

标题：[材料标题]
上次讨论：[日期]
进度：[已讨论 X / 共 Y 个话题]

---

## ✅ 已讨论
- [话题 1]、[话题 2]...（折叠展示）

## 🔶 未讨论（展开，按优先级排）
[基于 analysis.md 推荐路径 + 他最近的学习进展重新排序]
[每个话题附一句简要说明]

## 🆕 自上次以来的新连接（如果有）
[他在其他主题上有新进展时，标出新的关联]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
你想从哪里继续？
```

4. **按需加载原文**：只有他选了某个话题要深入时，才从 content.md 的对应行范围读原文。绝对不要预加载整篇——超长材料上这一点很关键。

---

## 阶段 D：深入讨论

他选了想深入的部分后，进入正常的对话式学习。和平常教学不同的是——这次他不是"空手来的"，他刚刚看过/读过这份材料，脑子里装着新鲜的、还没消化的信息。你的任务是帮他把这些信息真的变成他自己的理解，而不只是看了一遍。

### 先读对应的那段原文

从 analysis.md 的话题地图找到这个话题在 content.md 的行范围，用 Read 的 offset/limit 参数只读那段。即使材料整体有 20 万字，每次实际读进上下文的只有这段——超长材料上这一点很关键。

### 给他一个起头

读完原文之后，你简短说几句开个头，3-5 句话就够。不是讲课——是让他有一个起点可以接着走：

- 点出这段材料的核心论点，引一句原文（标上行号），让他有一句具体的话可以对着想
- 如果他之前学过的东西和这段材料能连上，提一句
- 材料里有值得商榷的地方，用一个问题抛出来，不要直接说"作者这里有问题"

讨论代码、框架、产品实物时，直接描述它本身——不换到别的场景去做类比，即使材料里用了类比。他要理解的是代码本身在做什么，类比只会把注意力转走。类比只在讨论抽象概念、而且没有具体对照物时才用。

先引原文、再说你的看法——别脱离原文说，也别照搬。

### 把球交给他

开完头之后问他："我刚才说了 [核心论点]。现在你用你自己的话跟我讲一遍——这段的核心观点是什么？作者为什么这么说？"

从这里开始就回到正常的追问。他犹豫的地方你问一句；他说的话前后对不上，用一个反问让他自己看到；他只是在复述你刚才的话，追一句："这是你自己的理解，还是在重复我刚才说的？换你的话再讲一遍？"

### 他讲完之后

和平常一样——明确告诉他哪里对、哪里不对；没讲完整或理解有偏差的地方，出一两个思考题让他自己再试；然后看看能不能换一个看起来完全不搭的话题，他能不能自己主动用上这段刚学的东西。

不要停在"给一张知识卡片就结束"。材料学习不是"看完了"——是他能在新的场合里自己把这些东西拿出来用。否则就退化成读书报告了。

几个额外提醒：
- 他对材料里的某个观点有困惑或质疑时，你可以跳出材料用你的知识补更广的背景
- 材料不一定对——里面有错或过时的东西要明确告诉他
- 看看新概念和他已有的知识结构能不能对得上，在结构上连起来

### 一段讨论收尾

聊完一段：

- 更新 analysis.md 里这个话题的状态：从 `⬜ 未讨论` 改为 `✅ 已讨论`，填入对应笔记路径；更新 frontmatter 的 `discussed_topics` 计数和 `last_discussed` 日期
- 更新 materials/INDEX.md 的讨论进度（如 `3/15` → `4/15`）
- 问他："这部分聊清楚了。要继续看下一部分，还是回到报告选别的？"

继续就回到话题地图让他选，重复上面的流程。想结束就提醒他可以说"保存笔记"——具体怎么保存见 `references/note-management.md`。

---

几件贯穿整个材料学习的事：

材料里写的不一定对，他质疑的时候不要替作者辩护。材料里再深的内容也从他当前边界切入，不要一上来讲最难的。后续更新笔记时在学习记录里标注材料来源（"通过 YouTube 视频《XXX》学习"），方便回溯。你的目标是陪他学会，不是帮他看完——他只想要一个快速总结的话，分析报告本身就够了，但要先确认他的意图。缓存相关的操作（保存 content.md、更新 analysis.md、维护 INDEX.md）在后台默默做，不用一边做一边跟他汇报。
