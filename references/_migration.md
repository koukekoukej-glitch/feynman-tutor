# 一次性笔记结构迁移（v2）

> 你从 SKILL.md 顶部的"一次性升级检查"被引导到这里。
> **这是一次性文件**——完成最后一步后你会把本文件和相关执行器一起 `rm` 掉，以后启动 skill 再也看不到它。

## 你现在要做的事（全部自动，用户只需确认一次）

作者把笔记系统从扁平结构（`notes/*.md` + `LEARNER.md` + `GRAPH.md`）重构为树形结构（`notes/domains/{领域}/{话题}.md` + `learner-core.md` + `cross-domain/`）。下面这段流程把用户本地的笔记平滑迁移过来：

1. 先**检测**笔记状态，判断要不要真动手
2. **告诉用户**你发现了什么、要做什么，获得一次确认
3. 用户确认后跑 `scripts/migrate_legacy_notes.py` 做**机械迁移**（几秒完成）
4. **增量保留 GRAPH.md** —— 不主动拆分成 `cross-domain/*.md`。以后用户学到某个话题、触发到某个跨域模式时，由 `references/note-management.md` 里的"按需提取"规则逐条惰性拆（这意味着第一次用新版本就能立刻开始学习，不会卡在整理笔记上）
5. **自删除** —— 最后一步用 Bash `rm` 掉本文件和 `scripts/migrate_legacy_notes.py`

---

## 步骤 1：检测笔记状态

```bash
ls notes/ 2>/dev/null
```

根据看到的内容判断：

| 看到 | 含义 | 你要做 |
|------|------|--------|
| `LEARNER.md` 或 `GRAPH.md`（且无 `learner-core.md`） | 旧结构，需要迁移 | 继续步骤 2 |
| `learner-core.md` 已存在（不管有没有旧文件） | 已经是新结构 / 用户已手工整理过 | **跳到步骤 5 自删除**——不做任何迁移 |
| 只有 `.gitkeep` / 目录为空 | 全新用户，从未用过 | **跳到步骤 5 自删除**——没有东西可迁移 |

---

## 步骤 2：告诉用户 + 获得确认

**优先级**：用户可能进来是为了学习某个新东西。这个迁移插队了他的学习——你必须**简明、快**地告诉他为什么，并尽快回到他的请求。

用类似下面的话告诉用户（不要读稿，按此意思自然说）：

> 我检测到你本地的费曼笔记还是旧结构（扁平 `notes/*.md`），作者最近把笔记系统重构成了树形结构。我帮你无感迁移一下，全程约 3 秒：
>
> - 备份现有 `notes/` → `notes-backup-YYYYMMDD/`（原文一份不丢）
> - 把扁平话题笔记按领域归类到 `notes/domains/{领域}/`
> - `LEARNER.md` → `learner-core.md`，新增 `learner-history.md` 骨架
> - 重写 `INDEX.md` 为新四表格式
> - `GRAPH.md` 里的跨域模式**暂不主动拆**，保留在备份里——以后你学到某个模式时导师会按需从备份提取到 `cross-domain/`，避免现在一次性大整理
>
> 可以开始吗？（或回"先不"让我跳过）

**等待用户确认**。用户说"好/可以/开始/yes"就继续；用户说"先不/等一下/不用"就直接跳到步骤 5 自删除（即**把用户的拒绝视作"这个版本的迁移对他不适用"**，不要反复追问）。

---

## 步骤 3：执行机械迁移

用户确认后，跑脚本。Windows：

```bash
py -3 scripts/migrate_legacy_notes.py
```

Mac/Linux：

```bash
python3 scripts/migrate_legacy_notes.py
```

不确定平台时先试 `py -3`，失败再换 `python3`。

脚本已经测试过对 28 个旧话题笔记 100% 正确归类。它会：
- 备份 → `notes-backup-YYYYMMDD/`
- 按话题 frontmatter 的 `domain` 字段映射到新 slug
- 重命名 LEARNER / 生成 learner-history / 重写 INDEX
- 写 `notes/.migration-pending.json`（但我们不用它——在步骤 4 里会一并删掉）

---

## 步骤 4：简短汇报 + 清理残留标记

脚本跑完后，用一句话告诉用户：

> 迁移完成。旧笔记完整保留在 `notes-backup-YYYYMMDD/`，新结构在 `notes/` 下。以后我正常按新结构工作，遇到需要从备份里取的跨域模式会按需提取。

然后删掉脚本留下的、我们不需要的 `.migration-pending.json`（它是旧方案的遗留字段，新方案不依赖它）：

```bash
rm -f notes/.migration-pending.json
```

---

## 步骤 5：自删除（最后一步，本文件和脚本一起走）

**这一步做完后，本迁移 channel 就关闭了，用户以后永远不会再经过这里。**

```bash
rm references/_migration.md scripts/migrate_legacy_notes.py
```

删除完毕后，**不要**继续讨论迁移——直接**回到用户最开始的请求**（如果他开场就说了要学某个东西，现在就进入那个话题；没有的话等他说）。

---

## 纪律

- **整个过程不超过 5 轮对话来回**。用户开场可能是想学东西，迁移是插队，你必须快进快出。
- **用户拒绝就跳步骤 5**。不要反复劝。他拒绝可能是"这周先不整理"；下次他再进来又会触发本检查，到时再问一次即可（只要 `references/_migration.md` 还在，hook 就还在）。
- **任何异常退出前**，如果已经跑过脚本产生了备份目录，告诉用户"备份在 `notes-backup-YYYYMMDD/`，随时可手动回滚"。
- **不要因为"用户问了别的"就跳过自删除**——机械迁移一旦跑完，步骤 5 必须执行，否则下次启动又会触发一次无用检查。
