# 费曼学习笔记管理

你从 SKILL.md 的收尾阶段被引导到这里，因为一轮学习对话即将结束，需要将学习成果持久化。

你的职责是从对话中提取有价值的学习信息，维护一套持久化的学习笔记。这套笔记是导师了解学生的核心工具——它让导师在下一次对话中能精准地知道学生的认知水平、学过什么、哪里有盲点，从而避免重复教已经会的东西，也不会跳过还没掌握的基础。

## 笔记存储

所有笔记存储在 `~/.claude/skills/feynman-tutor/notes/` 目录下，采用三层树形结构：

```
notes/
├── INDEX.md                      ← 导航索引（话题→领域映射 + 跨领域模式索引）
├── learner-core.md               ← 学习者核心模型（认知风格、盲区、有效策略）
├── learner-history.md            ← 认知成长轨迹（按时间线）
├── cross-domain/                 ← 跨领域模式（每个模式一个独立文件）
│   ├── 不可逆信息压缩.md
│   ├── 道-vs-术.md
│   └── ...
└── domains/{domain}/             ← 按领域组织
    ├── domain.md                 ← 领域框架 checklist + 前沿方向
    └── {topic}.md                ← 话题笔记
```

## 触发后的流程

### 第一步：收集信息

回顾当前对话，提取以下信息：

- **主题**：这轮对话学习了什么
- **领域**：属于哪个知识领域
- **起始水平**：对话开始时用户的认知水平
- **已掌握**：用户在本次对话中展示出扎实理解的概念（重点是用户自己讲出来的、讲对的部分，而不只是导师教过的）
- **已纠正的误区**：用户最初的错误理解，以及纠正后的正确理解
- **当前认知边界**：用户目前的理解停在哪里，下次应从哪里继续
- **待加强**：提到但未深入的概念，或用户表示还需进一步学习的内容
- **有效的类比**：对话中特别有效的类比或解释方式（这些是经过验证的——下次可以复用）
- **透镜化等级**（SKILL.md 阶段 2 + 阶段 3.5 的综合判定）：本次学习完成时，用户在该主题上达到了 L1-L6 的哪一级。判定依据见 SKILL.md 章节 2.3
- **失效边界**：本次讨论中触及的"这个概念在什么情况下不成立"的边界案例
- **透镜适用现象**：这个概念是观察什么类现象的透镜（用于 `future-call-task` 的现象描述部分）
- **下次自发调用任务**（`future-call-task`）：明确写出"下次遇到 Z 类情境时，主动检测 X 是否适用"的具体句子——Z 必须是具体可识别的情境类别，不是"任何相关场景"
- **本次检测到的自发调用**（仅当 SKILL.md 阶段 4 Step 1 扫描到时）：用户在讨论其他话题时自发引用了哪个已学概念，以及那次引用的 quality（accurate / misapplied）

### 第二步：检查已有笔记

读取 `~/.claude/skills/feynman-tutor/notes/INDEX.md`（如果存在）：
- 在话题索引中查找是否已有该主题的笔记，注意话题笔记现在存储在 `domains/{domain}/` 子目录下
- 如果已有相关主题的笔记，读取该笔记，准备**更新**而不是新建
- 如果是全新主题，确定其所属领域，准备在对应的 `domains/{domain}/` 目录下**新建**笔记

### 第三步：展示总结

向用户展示一份结构化的学习总结：

```
📓 学习笔记总结
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
主题：[主题名]
领域：[领域]
认知水平：[入门 / 进阶 / 掌握]
透镜化等级：L[n]（从 L[旧等级] 推进到 L[n]）
操作：[新建 / 更新]

✅ 已掌握：
- ...

🔄 已纠正的误区：
- ❌ [原来的错误理解]
- ✅ [纠正后的理解]

📍 当前认知边界：
...

📋 待加强：
- ...

💡 有效的类比/解释：
- ...

🔭 透镜画像：
- 失效边界：[本次触及的边界案例]
- 适用现象：[这是观察什么类现象的透镜]
- 下次自发调用任务：[下次遇到 Z 类情境时主动检测 X 是否适用]

🌱 本次自发调用（若有）：
- 在讨论 [A 话题] 时自发调用了 [B 概念]，quality：[accurate / misapplied]
  → 将同步更新 B 的笔记，virtuosity-level 提升为 L[n]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

然后问用户："这个总结准确吗？有需要补充或修改的地方吗？确认后我就保存。"

**等待用户确认。不要在用户确认之前写入任何文件。**

### 第四步：用户确认后保存

用户确认后（可能会要求修改，先按要求修改再保存），写入笔记文件。

**笔记文件格式：**

```markdown
---
topic: [主题名]
domain: [领域目录名，如 ai-agent-engineering]
level: [beginner/intermediate/advanced]
virtuosity-level: L[1-6]                    # 透镜化等级，见 SKILL.md 章节 2.3
last_updated: [YYYY-MM-DD]
last_verified_at: [YYYY-MM-DD]              # 上次透镜化等级被直接验证的日期（通过阶段 3.5 探针或自发调用）
status: [in-progress/completed]
cross-domain-patterns:
  - [相关的跨领域模式文件名，如 道-vs-术]
  - [...]
related-topics:
  - [相关话题文件名，如 prompt-engineering-methodology]
  - [...]
spontaneous-calls:                          # 自发调用日志（跨会话回填）
  - date: YYYY-MM-DD
    context: "讨论 [A 话题] 时自发引用"
    quality: accurate                       # accurate | misapplied
future-call-task: "下次遇到 [Z 类情境] 时，主动检测 [X 概念] 是否适用"
---

## 认知画像
[一段话描述用户在这个主题上的整体认知状态，用自然语言写，像是一个老师在跟同事交接学生情况。**如果 virtuosity-level ≥ L4，加一句概括这个等级的证据**——比如"在 [日期] 主动批评了'A 类比在 X 情境失效'"。]

## 已掌握
- [概念1]：[用用户自己的语言描述的理解]
- [概念2]：...

## 已纠正的误区
- [误区1]：原以为... → 实际上...

## 透镜画像
- **失效边界**：[本概念在什么情况下不成立——讨论中触及的边界案例]
- **透镜适用现象**：[这是观察什么类现象的透镜]
- **等级跃迁记录**：[YYYY-MM-DD] L[旧] → L[新]，触发事件：[简述]

## 当前认知边界
[描述用户目前的理解停在哪里，下次对话应该从这里继续]

## 待加强
- [概念/技能]：[为什么需要加强]

## 有效的类比
- [类比1]：[适用场景]

## 学习记录
- [YYYY-MM-DD]：[本次学习的简要摘要 + virtuosity-level 变化]
```

**更新已有笔记时的规则：**
- 「已掌握」：追加新掌握的内容，保留之前的
- 「已纠正的误区」：追加新发现的误区
- 「当前认知边界」：更新为最新的边界位置
- 「待加强」：如果某项已经在本次学会了，移到「已掌握」；追加新的待加强项
- 「学习记录」：追加本次记录
- 「认知画像」和 `level`：根据整体进展更新
- `virtuosity-level`：按 SKILL.md 章节 2.3 的判据更新。**只允许在"本次有新证据"时更新——不要因为用户"看起来更熟"就升级。只有阶段 2 诊断或阶段 3.5 探针明确给出等级证据时才动这个字段。**
- `last_verified_at`：仅在 `virtuosity-level` 被直接验证（阶段 3.5 探针触发 或 检测到自发调用）时更新。光是用户"又学了一次"不算验证。
- 「透镜画像 · 等级跃迁记录」：每次 `virtuosity-level` 变化时追加一行
- 「透镜画像 · 失效边界 / 适用现象」：如本次讨论产生新边界或新适用现象，追加
- `future-call-task`：覆盖为最新版本（只保留最当前的一条，不累积）

**从历史备份按需提取（仅在 `notes/` 同级存在 `notes-backup-*/` 目录时激活，否则本节完全静默）：**

笔记系统有"加载时按需惰性提取"的机制——当 `notes/` 旁边存在备份目录（形如 `notes-backup-YYYYMMDD/`，可能因历史结构升级而产生）时，有些信息还在备份里没搬到新结构。按以下规则增量提取：

**情形 A——话题笔记 frontmatter 缺跨域指针**

加载当前话题笔记时，如果发现 frontmatter **没有 `cross-domain-patterns` 或 `related-topics` 字段**（完全缺失 ≠ 空列表），在本次笔记保存阶段顺手补上：

1. 读备份目录下的 `GRAPH.md`
2. 搜索所有提到当前话题名或当前话题文件 stem 的条目——典型出现在每个模式条目的 `- 涉及：...` 行里
3. 命中的每个加粗模式标题（`**模式名**`）→ 加入本话题 frontmatter 的 `cross-domain-patterns`
4. 命中条目里一起提到的其他话题名——在 `notes/domains/**/*.md` 下找匹配 → 加入 `related-topics`
5. 补完后即使命中为空，也**写入空列表 `[]`**（标记"已检查过"），避免下次加载再做一次无用扫描

**情形 B——指针指向的 cross-domain 模式文件不存在**

加载话题笔记的 `cross-domain-patterns` 指针时，如果某个指向的 `cross-domain/{模式}.md` 文件不存在，但备份的 `GRAPH.md` 里有该模式的条目——**只把这一个模式**从备份提取出来写到 `cross-domain/` 下，同时更新 `INDEX.md` 的跨领域模式索引表追加一行。

不要一次性提取所有模式。**用户学到哪个、用到哪个，才提取哪个**——这是按需增量，不是批量整理。

**提取纪律**：
- 只处理当前话题直接相关的内容。其他话题有自己的加载时刻再处理自己。
- 所有写入合并进本次笔记保存流程，**不做独立写文件动作**
- 备份里可能有笔误——**不就地纠正**。搬家不改家具，保留原文让用户在重温时自然察觉
- 所有 `notes-backup-*/` 都是只读参照，**不写入、不删除**——用户随时可以手动删除整个备份目录，此后本节自然永久静默

**跨话题回填（由 SKILL.md 阶段 4 Step 1 触发）：**

如果本次学习中 SKILL.md 阶段 4 Step 1 扫描到了对**其他话题**已学概念的自发调用，按下列规则更新那个**被调用的话题**（非当前学习主题）的笔记：
- 如果在无关话题中自发调用且分析成立 → 将该话题的 `virtuosity-level` 更新为 L6（若原本低于 L6），更新 `last_verified_at`
- 如果轻推后调用 → 更新为 L5（若原本低于 L5）
- 在该话题的 `spontaneous-calls` 追加一条记录
- 在「学习记录」追加一条"[日期]：在讨论 [A 话题] 时被自发调用，quality [accurate/misapplied]"
- **不修改该话题的其他字段**——只做等级验证和日志追加

**更新 learner-core.md frontmatter（每次必做）：**
- `sessions_count`：+1
- `last_updated`：当天日期
- `topics_studied`：追加本次话题的文件名（不含 .md）

**更新 learner-history.md（每次必做）：**
在文件末尾追加本次学习的成长记录条目。格式：`- **YYYY-MM-DD**（第 N 轮，主题名）：[一段话描述本次学习的认知路径和标志性特征]`。同时更新 frontmatter 的 `sessions_count` 和 `last_updated`。

**更新 INDEX.md：**

INDEX.md 采用四表结构（元文件、领域目录、话题索引、跨领域模式索引）。新建话题时在话题索引表中追加一行；新建领域时在领域目录表中追加一行并更新话题数。参考现有 INDEX.md 格式。

**新建话题时的文件放置：** 话题笔记放在 `domains/{domain}/` 目录下。如果是全新领域，先创建 `domains/{domain}/` 目录和 `domain.md`，格式如下：

```markdown
---
type: domain
domain: {domain-slug}
topic-count: 1
---

### {领域人类可读名称}

**[子领域/方向名]**
- ✅ [本次学习的话题/概念]
```

**INDEX.md 更新规则：**
- 新建话题 → 话题索引追加一行，对应领域的话题数 +1
- 新建领域 → 领域目录追加一行（话题数初始为 1），话题索引追加一行
- 新建跨领域模式（第六步中）→ 跨领域模式索引追加一行

**frontmatter 指针映射规则：** `cross-domain-patterns` 和 `related-topics` 列表中的值必须与对应目录下的文件名（不含 .md 后缀）完全一致。加载时直接拼接路径：`cross-domain/{值}.md` 或 `domains/{domain}/{值}.md`。

### 第五步：确认保存

保存完成后，简短告知用户笔记已保存，然后**立即进入第六步**。

### 第六步：认知地图更新

保存主题笔记后，更新分布式认知地图。认知地图不再是单一文件，而是分布在三个层级中。

**流程：**

1. **前置门控**：检查以下两个信号，**至少命中一个**才继续扫描：
   - **用户主动联想**：对话中用户（不是导师）主动提到了其他领域的概念或经验
   - **模式复现**：用户学到的某个机制/原则，在已有的 `cross-domain-patterns` 中已有记录，且本次对话为该模式提供了新的具体实例
   - **命中 ≥ 1 个** → 继续下方步骤 2-3 扫描
   - **两个都没命中** → 跳过步骤 2-3，直接到步骤 4。步骤 5 展示时只展示领域框架更新部分，省略跨领域连接和学习者模型板块

2. **扫描跨领域连接**——将本次学到的新内容与已加载的跨领域模式和领域知识做对比：
   - **已有模式扩展**：本次学习是否为某个已有的 `cross-domain/{pattern}.md` 提供了新实例或新维度？
   - **新模式发现**：本次学到的某个机制/原则，是否在 ≥2 个已学领域中出现？如果是，可能需要创建新的跨领域模式文件
   - **张力/矛盾**：本次学到的"最佳实践"在另一个领域中是反模式吗？
   - **认知迁移**：对话中有没有从旧领域迁移过来的特别有效的思维方式？

3. **扫描学习者模型变化**——对比本次对话中的认知模式与 `learner-core.md` 的记录：
   - 有没有新的跨主题认知风格（需 ≥2 个主题验证）？
   - 有没有新的习惯性盲区？
   - 有没有新的有效策略被验证？
   - 已记录的盲区是否在本次对话中被证明已克服？（用户能自如处理之前标记为盲区的模式）→ 在 learner-core.md 中标记为「已克服」而非删除，保留成长记录
   - 成长轨迹有没有标志性事件值得记录？

4. **更新领域框架**——在 `domains/{domain}/domain.md` 中，将本次学习覆盖的新概念标记为 ✅

5. **向用户展示发现**——只展示有实质意义的发现（不要为了填充而捏造连接）：

```
🔗 认知地图更新
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📌 跨领域连接：
- [已有模式扩展/新模式]：[简述]
  涉及：[主题A] ↔ [主题B]

🧠 学习者模型：
- [新观察到的模式/盲区/策略]

📊 领域框架：
- [领域名] checklist 更新

确认后我就更新。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**等待用户确认。**

6. **确认后分布式写入：**
   - **跨领域模式**：更新已有的 `cross-domain/{pattern}.md`（追加新实例/维度），或创建新文件。新建时需要加 frontmatter（type、category、related-domains）
   - **话题笔记 frontmatter**：如果发现了新的跨领域连接，在当前话题笔记的 `cross-domain-patterns` 列表中追加指针。如果关联到新的话题，更新 `related-topics`
   - **学习者核心模型（内容体）**：如果有新的元认知模式 → 更新 `learner-core.md` 的认知风格/盲区/有效策略章节（frontmatter 已在第四步更新）
   - **领域框架**：更新 `domains/{domain}/domain.md` 的 checklist
   - **INDEX.md**：如果新增了跨领域模式，更新跨领域模式索引表（话题索引已在第四步更新）

**关键原则：**
- **每次只提 1-2 个高质量连接**，质量远比数量重要。如果本次确实没有值得记录的跨主题连接，坦诚告知并跳过。
- **连接必须附带"为什么"**——不只是"A 和 B 相关"，而是"A 中的 X 机制和 B 中的 Y 机制具有同构性，因为……"
- **学习者模型只记录跨主题模式**——单一主题内的认知状态留在话题笔记的"认知画像"中。
- **frontmatter 指针是导航的关键**——每次更新笔记时，确保 cross-domain-patterns 和 related-topics 字段是最新的，这决定了下次加载时的探索路径。

### 第七步：写入完整性验证

**IMPORTANT：所有写入完成后，逐项核对以下清单。每项标注 ✅ 或 ⬜（不适用时标注 N/A 并说明原因）。有未完成项不得告知用户"已完成"。**

| # | 文件 | 更新内容 | 条件 |
|---|------|---------|------|
| 1 | `domains/{domain}/{topic}.md` | 话题笔记（新建或更新） | **每次必做** |
| 2 | 当前话题笔记 frontmatter | `virtuosity-level`, `future-call-task` 更新；`last_verified_at`（仅在等级被直接验证时） | **每次必做** |
| 3 | 当前话题笔记「透镜画像」 | 失效边界 / 适用现象 / 等级跃迁记录 | **每次必做** |

## 重要原则

- **用用户自己的语言记录。** 笔记的读者是下一次对话中的费曼导师，目的是让导师快速了解这个学生的实际认知状态。忠实反映用户的理解方式，比用教科书术语更有价值。
- **认知水平基于对话中的实际表现判断。** 用户能流畅教出来的才算"已掌握"，导师讲过但用户没有回应验证的只能算"待加强"。
- **一次对话可能涉及多个主题。** 如果对话跨了多个领域，为每个主题分别创建/更新笔记，并逐个展示总结。
- **笔记要精炼。** 每个条目用一两句话概括，不要复制对话原文。笔记是索引，不是抄本。
